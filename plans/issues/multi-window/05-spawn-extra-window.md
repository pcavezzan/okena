## What to build

The first user-visible end-to-end demo of multi-window. A new action — call it `NewWindow` — that creates an additional viewport onto the Workspace, spawned with an empty filter (every current project hidden) and cascade-offset bounds.

Wiring:

- A `NewWindow` action type registered in the keybindings module, bound to Cmd+Shift+N.
- A menu entry "Window → New Window" (introduce a top-level "Window" menu in `set_app_menus` if one doesn't exist).
- A command palette entry labeled "Window: New".
- The action handler mutates `WorkspaceData.extra_windows`: pushes a new `WindowState` whose `hidden_project_ids` is a snapshot of every current project ID, `folder_filter: None`, `project_widths: empty`, `folder_collapsed: empty`, `os_bounds: cascade-offset from spawning window's bounds`.

`Okena` observes the workspace and, on detecting a new entry in `extra_windows`, opens an OS window via `cx.open_window(...)` with the persisted `os_bounds` (cascade fallback if absent), wraps a fresh `WindowView` (with the new `WindowId`), inserts the entity into `Okena::extra_windows` map. The new window shares all existing globals (settings, theme, terminals registry, PTY manager, git watcher, service manager, remote manager).

Empty grid in the new window shows a placeholder ("No projects in this window — click a project in the sidebar to show it here") so the empty state doesn't look broken.

Action routing (focused-window scope) is already inherited because GPUI dispatches keybinding actions to the focused element's hierarchy. Verify that triggering `NewWindow` from within an extra spawns yet another extra (cascading fine), and that overlays / palette / settings dialogs open inside the focused window.

After this slice: pressing Cmd+Shift+N opens a fresh empty window. The user can click any project in its sidebar — focus zoom (already per-window from slices 02/03) reveals it. The user can manage filter state per window via existing UI (folder filter, hide/show via context menu).

## Acceptance criteria

- [x] `NewWindow` action exists, bound to Cmd+Shift+N, listed in the menu and command palette.
- [x] Triggering the action from the main window: a second OS window opens, offset (e.g. +30,+30) from the spawning window's bounds.
- [ ] The new window's projects-grid is empty; sidebar lists every project (just like main); chrome (title bar, status bar, toasts, overlay manager, command palette, theme selector) all work and are scoped to that window.
- [x] An empty-grid placeholder is rendered when `hidden_project_ids` covers every project in the workspace.
- [x] Triggering `NewWindow` again opens a third window (no artificial cap).
- [x] Triggering the action from an extra window (rather than main) spawns the new window with cascade offset from that extra.
- [ ] Spawning a window does NOT trigger save churn beyond the existing debounced auto-save — the new entry is written via the same `save_pending` mechanism.
- [ ] Per-window FocusManager works in isolation: zooming a project in W2 leaves W1 unzoomed.
- [ ] Per-window column widths: dragging a divider in W2 changes only W2's `WindowState.project_widths` for that project; W1's widths unchanged.
- [ ] Per-window folder filter: selecting a folder filter in W2 leaves W1's filter alone.
- [ ] Per-window folder collapse: collapsing a folder in W2's sidebar leaves W1's expansion state alone.
- [ ] Same terminal rendered in W1 and W2 (after unhiding the project in W2 via focus zoom) updates in both simultaneously when output arrives — verifies the slice 04 fan-out under real load.
- [ ] CLI fallback: invoking `okena open <path>` while no window is focused targets main; while W2 is focused, targets W2.
- [x] `cargo build` and `cargo test` both green.
